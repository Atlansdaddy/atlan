# Atlan — technical review packet (for AI critique)

> Paste this to a frontier model (ChatGPT / Claude / Gemini / Grok) and ask it to **critique, not praise** — find the flaws, the risky assumptions, the security holes, the places the design will break at scale. The repo is public and browsable: **https://github.com/Atlansdaddy/atlan** (Apache-2.0). Be adversarial. I want the truth.

## What it is
Atlan is a **personal AI build cockpit that runs entirely on a phone** — a single web app (served from Node inside Ubuntu proot under Termux on a Samsung S24 Ultra, also runs on any Linux/macOS) that turns the phone into a real dev surface. One page gives you: Claude Code (via the Claude Agent SDK) with permission cards; other agent CLIs (Codex, Gemini) headless; local models (llama-server) and cloud "brains"; a live preview sandbox that pipes console errors back to the agent; a real tmux terminal; a code editor (CodeMirror, 122 languages); message attachments (images→vision, audio/video→multimodal delegation); an autonomous agent **fleet** with hard token budgets and profiles; a **worker hierarchy** (orchestrator-workers: cheap local model does bulk, escalates to frontier on failure, deterministic checkers gate each step, human gate at tier-3 risk); a **Persona+ builder** that compiles a persona framework into typed tools with code-graded checkers; one-button APK builds; and health/security dashboards. Loopback-only by design.

## Stack + notable engineering decisions (and the *why* — critique these)
1. **No-build vanilla front-end.** Deliberately no bundler, no framework, no app service worker — fewer moving parts to break in proot, and it structurally kills the stale-service-worker class of bug. *Critique: does this cost more than it saves as the app grows?*
2. **Everything fragile lives behind an adapter + a Doctor check.** Every proot-boundary hack (JDK, Android SDK, the qemu-aapt2 shim that runs x86 aapt2 on ARM, the claude binary, tmux) has a green/red health check, so a Termux update breaks one visible thing, not the app.
3. **Fleet profiles are hard walls, not requests.** Scout = read-only (write/exec tools *stripped* at the SDK level via `disallowedTools`, not merely denied — proven live). Budgets **halt** (checked between steps; a single step can overshoot the cap, but no unbounded spend). *Critique: the between-steps overshoot; the fact that `canUseTool` alone was NOT a gate (settings allowlists + sandboxed-"safe" Bash auto-approved past it) — we fixed with `settingSources:[]` + `disallowedTools`, but is that airtight?*
4. **Deterministic checkers, never a model grading a model.** Persona+ structured commands compile to JSON-schema-constrained output; checkers are code assertions (enum, range, regex, subset-of-input-variable, arithmetic via a hand-written safe evaluator with no `eval` reachable). A tiered taxonomy: tier-1 format failures die at constrained decoding, tier-2 referential/arithmetic at the checkers, tier-3 semantic-but-valid goes to a human. *Critique: where does "code grades the model" break down?*
5. **Password auth + httpOnly session; header bearer for automation.** scrypt-hashed password, `SameSite=Strict` cookie surviving restarts, throttled logins; never a token in a URL (we made that mistake and reverted it). *Critique: the auth model for a loopback app; the automation bearer; is there a CSRF or session-fixation angle?*
6. **Honest cost accounting.** On a Claude subscription there's no per-token charge, so the "$" is labeled "≈API-equiv" (a work gauge, not a bill); tokens are the real metered currency, which is what budgets protect. Cache-read tokens are shown separately. *Critique: is this honest framing actually correct?*
7. **Provider ToS posture.** We drive the *official first-party CLIs* (the sanctioned path) and never proxy subscription OAuth tokens into our own client (the thing that got other tools' users banned). The Agent-SDK-on-subscription path is acknowledged as a gray area (docs say devs "should" use API keys — advisory, not the binding prohibition, which is "on behalf of other users," which we don't do). *Critique: is our ToS reasoning sound, or are we rationalizing?*

## Security posture (please attack this)
- Loopback-only; password + session gate on `/api`, `/apk`, WS. SSRF-guarded preview + harness (loopback-only, tested against decimal/hex/IPv6/rebind bypasses). Secrets AES-256-GCM at rest. Path guards realpath-check for symlink escape (a fleet scout auditing our own code found the original symlink gap — we fixed it).
- **Known, documented gaps:** the OS-level Bash sandbox (bubblewrap/Landlock) **cannot run in proot** (no namespaces — tested), so builder/verifier Bash is gated by *tool profile*, not OS-confined within the machine (proot still confines everything to the Termux app sandbox). The preview proxy is unauthenticated loopback. Cloud engines see prompts under their own ToS.
- *Critique: given loopback + proot + password, what's the realistic threat model, and where are we actually exposed? Is the "run untrusted work on a native host for the real sandbox" answer good enough?*

## Test discipline
152 automated tests across 10 suites (unit, function+data-store-durability, connection/WS/PTY, security/pentest, adversarial, worker-hierarchy, attachments, code-editor, UI/UX headless-Chromium, tour/onboarding), run on a **throwaway instance** so they never touch the live cockpit. Verbatim output in `docs/RECEIPTS.md`. We also spun **black-box contextless agents** (no source access) to attack it — they found real bugs our own tests missed (silently-dropped checkers, Infinity-arithmetic passing vacuously, concurrency spend-multiplication). *Critique: what would a determined adversary still find? What's under-tested?*

## The research-backed parts (critique the conclusions)
- **Token optimization** (deep-research, verified): input tokens dominate agentic-coding cost ~154:1; more tokens ≠ more accuracy (peaks at intermediate spend). We *measured our own* caching: a warm/resumed turn cut fresh tokens 39,740→1,218 (−97%). Model/tier routing is a first-order lever (the hierarchy). We concluded: image-as-context has no 2026 evidence (avoid), compaction is lossy-on-numbers (never on checker data), grep+vector hybrid beats either alone. *Critique: are these conclusions right?*
- **The vault (designed, not yet built)** — `docs/VAULT-DESIGN.md`: a grep+vector-hybrid, on-device SQLite (FTS5 + sqlite-vec) knowledge base with Mem0-style ADD/UPDATE/MERGE dedup, micro/macro/major tiers, and a **gated self-repair** stage (detect→propose→checker gate→verify-in-worktree→human gate→rebuild; no recursion, rate-limited, circuit breaker, kill switch, off by default). *Critique: will the dedup/consolidation actually prevent knowledge sprawl? Are the self-repair gates sufficient to prevent runaway self-modification?*

## The three questions I most want torn apart
1. **Security:** given loopback + password + proot (no OS Bash sandbox), what's the real exposure, and is the "native host for untrusted work" mitigation honest or a cop-out?
2. **The self-repair vault gates:** are checker-gate + worktree-verify + human-gate + no-recursion + circuit-breaker actually enough to make autonomous self-repair safe, or is there a hole?
3. **Sustainability:** as a solo-built, no-framework, phone-hosted app with 20 tabs of features — what breaks first at scale, and what would you cut?
